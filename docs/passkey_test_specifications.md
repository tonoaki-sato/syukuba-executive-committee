# 保土ケ谷宿場まつり実行委員会 実務管理総合システム パスキー（WebAuthn）認証機能 テスト仕様書

本書は、本システムに導入されたパスキー（WebAuthn）による認証機能（登録、ログイン、管理、リカバリー等）を対象としたテストケースおよび検証手順を定義する。

---

## 1. テスト概要

パスキー認証は従来のパスワード認証と異なり、ブラウザの WebAuthn API およびデバイスの生体認証（指紋・顔認証・PIN）を介して動作する。
本テスト仕様書では、バックエンドの検証ロジックを保証する**「自動テスト（Featureテスト）」**と、実際のUI/UXおよびデバイス連携を担保する**「手動テスト（ブラウザ/実機検証）」**の双方について定義する。

---

## 2. 自動テスト仕様（API・ロジック検証）

パスキー認証および登録に関わるバックエンドのエンドポイント・ビジネスロジックの検証ケース。
（想定テストクラス名：`tests/Feature/WebAuthnAuthenticationTest.php`）

### 2.1 テスト対象エンドポイント
- **パスキーログイン用API**：
  - `POST /webauthn/login/challenge` (チャレンジ生成)
  - `POST /webauthn/login/verify` (認証検証・ログイン)
  - `POST /webauthn/login/check` (パスキー登録判定)
- **パスキー登録用API**：
  - `POST /webauthn/register/challenge` (登録用チャレンジ生成)
  - `POST /webauthn/register/verify` (公開鍵の検証・保存)

## 3. 手動テスト仕様（UI・実機・運用検証）

WebAuthn はセキュリティ制約（HTTPS必須、localhost例外、同一オリジン等）があるため、実際のブラウザ環境で動作検証を行う。

### 3.1 テスト前提条件
- **接続環境**:
  - 本番/ステージング環境の正規 HTTPS ドメイン（`https://www.syukuba.home`）または開発環境の `http://localhost`。
- **検証デバイス・ブラウザ**:
  - 指紋認証（Touch ID/Windows Hello）または顔認証、または外部セキュリティキー（YubiKeyなど）が利用可能なデバイス。
  - Google Chrome / Safari / Edge / Firefox の最新版。

### 3.2 手動テストシナリオ

#### シナリオ 1: 新規メンバーの追加からパスキー初回登録の流れ
1. **システム管理者**でログインし、会員追加画面（`/admin/users/create`）から「パスワード自動生成」をONにして新規会員 `test_member@example.com` を追加する。
2. 画面に表示される「一時ログインパスワード」と「パスキー設定用ワンタイムURL」をコピーして控える。
3. 管理者アカウントから**ログアウト**する。
4. コピーしたワンタイムURL（例: `https://www.syukuba.home/passkey/register/xyz...`）にブラウザでアクセスする。
5. **期待値**: 未ログイン状態のため、ログイン画面（`/login`）へリダイレクトされること。
6. `test_member@example.com` と控えておいた「一時ログインパスワード」を入力してログインする。
7. **期待値**: ログイン成功直後、パスキー登録専用画面（`auth.passkey_register` テンプレート）が自動表示されること。
8. 任意のデバイス名を入力し、「このデバイスを登録する」ボタンをクリックする。
9. **期待値**: ブラウザの生体認証（またはセキュリティキー）ダイアログが立ち上がること。
10. デバイスの指紋/顔/PINを入力して登録を完了させる。
11. **期待値**: 登録成功のメッセージが表示され、ワンタイムURLが使用不可になる（再アクセス時にエラーまたはマイページへの遷移となる）こと。

#### シナリオ 2: パスキー優先（Conditional UI / Autofill）によるログイン
1. すべてのセッションをログアウトし、ログイン画面（`/login`）を開く。
2. 「メールアドレス」の入力欄をクリック（フォーカス）する。
3. **期待値**:
   - 入力欄の下部またはキーボード上部に、登録済みのパスキー（例: `test_member@example.com` 用の資格情報）が自動提案されること。
4. 提案されたアカウントを選択する。
5. **期待値**:
   - 生体認証ダイアログが起動し、認証が成功すると、メールアドレスやパスワードをタイピングすることなく、即座にダッシュボード画面へログインできること。

#### シナリオ 3: 通常ログイン導線からの自動パスキー起動とパスワードフォールバック
1. ログアウトした状態でログイン画面を開く。
2. メールアドレス入力欄に、パスキー登録済みの `test_member@example.com` を手動入力し、「次へ」を押す。
3. **期待値**:
   - パスワード入力欄が表示される前に、ブラウザの生体認証ダイアログが自動的に起動すること。
4. 認証ダイアログの「キャンセル」または「スキップ」ボタンを押し、パスキー認証を意図的に中断する。
5. **期待値**:
   - パスワード入力画面にフォールバックし、正しいパスワードを入力することでログインできること。

#### シナリオ 4: 複数デバイスのパスキー追加登録
1. パキー登録済みのユーザーでログインし、マイページ（`/mypage`）へ移動する。
2. 「パスキーデバイス一覧」または管理者用画面より「新しいパスキーの追加」を行う（※運用上、別デバイスを登録する場合は、管理者が対象ユーザー用の追加セッションURLを発行し、そのURLに対象ユーザーでログインした状態でアクセスして登録する）。
3. **期待値**:
   - 2台目（例：スマホからPCなど）のデバイスで同様に生体認証が起動し、問題なく登録できること。
   - ログイン時の Conditional UI および自動起動で、どちらのデバイスからでもログインができること。

#### シナリオ 5: 管理者によるパスキーの無効化（削除）とログイン拒否
1. **システム管理者**でログインし、会員一覧から対象ユーザーの「詳細プロフィール」（`/users/{id}`）を開く。
2. 「登録済みパスキーデバイス」リストから、特定のパスキーの「削除」ボタンをクリックする。
3. **期待値**:
   - 削除完了のフラッシュメッセージが表示され、リストから対象デバイスが消去されること。
4. 管理者をログアウトする。
5. 削除したデバイスを使用して、パスキーログインを試行する。
6. **期待値**:
   - ログイン画面のオートフィルに現れなくなる、または認証を試みても「登録されていないパスキーです」等のエラーになりログインできないこと。
