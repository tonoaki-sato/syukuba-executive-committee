# 保土ケ谷宿場まつり実行委員会 実務管理総合システム 管理者用ユーザー直接追加機能 テスト仕様書

本書は、システム管理グループ（管理者）による「管理者用ユーザー直接追加機能」を対象としたテストケースおよび検証手順を定義する。

---

## 1. テスト概要

本テストでは、アクセス制限（セキュリティの担保）、入力値検証（バリデーション）、データ整合性（データベース保存結果および関連レコード生成）、ならびに管理者向けのUI表示（一時パスワードおよびワンタイム登録URLの発行と表示）の正常動作を保証する。

テストは、**「自動テスト（Featureテスト）」**と**「手動テスト（ブラックボックステスト）」**の双方を用いて実施する。

---

## 2. 自動テスト仕様（Featureテスト）

本機能のために実装されたFeatureテスト（[AdminUserCreationTest.php](file:///opt/project/syukuba-executive-committee/tests/Feature/AdminUserCreationTest.php)）のテストケース一覧。

### 2.1 テストクラス
- 物理ファイル：`tests/Feature/AdminUserCreationTest.php`
- テストコマンド：`php artisan test --filter=AdminUserCreationTest`

### 2.2 テストケース詳細

| No | テストケース名 / メソッド名 | 検証内容（テスト条件・操作） | 期待値（結果） |
| :--- | :--- | :--- | :--- |
| 1 | `test_guest_cannot_access_creation_routes` | 未ログインのゲストが以下のURLにアクセスする：<br>・`GET /admin/users/create`<br>・`POST /admin/users` | ログイン画面（`/login`）へリダイレクトされること。 |
| 2 | `test_general_user_cannot_access_creation_routes` | ログイン済みの「一般会員」権限ユーザーが以下のURLにアクセスする：<br>・`GET /admin/users/create`<br>・`POST /admin/users` | `403 Forbidden`（アクセス拒否）となること。 |
| 3 | `test_admin_can_access_creation_form` | ログイン済みの「システム管理」権限ユーザーが追加フォーム（`GET /admin/users/create`）にアクセスする。 | ステータス `200 OK` となり、追加フォーム画面が正常に表示されること。 |
| 4 | `test_admin_can_store_user_with_auto_generated_password` | 管理者が「パスワード自動生成」をONにして新規会員を追加する。 | ・会員一覧にリダイレクトされること。<br>・`status = active` でDB保存されること。<br>・`approved_by` に管理者IDが記録されること。<br>・当年度の在籍レコード (`UserYear`) が自動生成されること。<br>・24時間有効なパスキー登録セッションがDBに作られること。<br>・リダイレクト後のセッション内に一時パスワードとパスキー登録URLが保持されていること。 |
| 5 | `test_admin_can_store_user_with_manual_password` | 管理者が「パスワード自動生成」をOFFにして、パスワードを手動で指定して新規会員を追加する。 | ・指定したパスワードでハッシュ化されてDBに保存されること。<br>・そのパスワードでログインが可能となること。 |
| 6 | `test_store_validation_fails_for_invalid_data` | バリデーションエラーとなる不正なデータを送信する：<br>1. 「かな」にひらがな以外が含まれる。<br>2. 既に登録済みのメールアドレスを指定。<br>3. 手動パスワード設定時の確認用入力との不一致。 | データベースに保存されず、それぞれ対応するバリデーションエラーメッセージが返却されること。 |

## 3. 手動テスト仕様（UI・運用検証）

テスト環境で管理画面を操作して実施する手動テストシナリオ。

### 3.1 テスト前提条件
- **テスト用ブラウザ**: Google Chrome または Safari（生体認証に対応したブラウザを推奨）
- **検証アカウント**: システム管理者権限を持つアクティブなアカウント

### 3.2 テストシナリオ一覧

#### シナリオ 1: 画面導線およびアクセス制限の確認
1. **システム管理者**としてログインする。
2. 「会員一覧・パスキー管理」画面（`/admin/users`）へアクセスする。
3. 画面右上に「➕ 新規会員を直接追加」ボタンが表示されていることを確認する。
4. ボタンをクリックし、ユーザー追加画面（`/admin/users/create`）が表示されることを確認する。
5. 一度ログアウトし、**一般会員**のアカウントでログインし直す。
6. ブラウザのアドレスバーに直接 `/admin/users/create` を入力してアクセスを試み、`403 Forbidden` エラー画面が表示されることを確認する。

#### シナリオ 2: パスワード自動生成による直接追加とURLコピーの確認
1. 再度**システム管理者**としてログインし、`/admin/users/create` へ移動する。
2. 入力フォームに以下の通り入力する：
   - 氏名（漢字）：`宿場 太郎`、氏名（かな）：`しゅくば たろう`
   - メールアドレス：`shukuba_test@example.com`
   - **初期パスワードを自動生成する**：チェックを **ON** にする（パスワード入力欄が非表示になることを確認）
   - 本業・職業：`自営業`、LINE表示名：`しゅくば太郎@LINE`
   - 得意分野：`電気工事`
   - 初期ロール：`一般会員` にチェック
3. 「ユーザーを登録」ボタンをクリックする。
4. 登録が完了し、会員一覧画面（`/admin/users`）にリダイレクトされ、画面上部に「会員「宿場 太郎」を直接追加しました」という青いカード（表示エリア）が現れることを確認する。
5. カード内に **「初期ログイン用パスワード（赤字のコード）」** と **「パスキー設定用ワンタイムURL」** が表示されていることを確認する。
6. 「📋 URLをコピー」ボタンをクリックし、ボタンが「✅ コピー完了！」に変化し、クリップボードに正しい登録用URLがコピーされることを確認する。

#### シナリオ 3: 追加されたユーザーでの初回ログインとパスキー登録の確認
1. システム管理者をログアウトする。
2. シナリオ2で発行された **「初期ログイン用パスワード」** と **「パスキー設定用ワンタイムURL」** を準備する。
3. コピーした「パスキー設定用ワンタイムURL」（例: `/passkey/register/{token}`）にブラウザでアクセスする。
4. 未ログイン状態であるため、ログイン画面（`/login`）へリダイレクトされることを確認する。
5. 追加されたユーザーのメールアドレス（`shukuba_test@example.com`）を入力し、「次へ」を押す。
6. パスワード入力欄が表示されるため、控えておいた **「初期ログイン用パスワード」** を入力し、「ログイン」を押す。
7. ログイン成功後、そのまま「パスキー（生体認証）設定画面」が表示されることを確認する。
8. 任意のデバイス名を入力し、「このデバイスを登録する」を押す。
9. ブラウザの生体認証ダイアログ（Windows Hello, Touch ID, Face IDなど）が起動し、登録が完了すると「登録完了」の緑色のメッセージが表示されることを確認する。
