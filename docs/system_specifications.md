# 保土ケ谷宿場まつり実行委員会 実務管理総合システム 仕様書

本書は、「保土ケ谷宿場まつり実行委員会」における実務を管理するための総合システムの仕様書である。

---

## 1. システム概要
本システムは、保土ケ谷宿場まつり実行委員会（以下、実行委員会）の日常的な実務運営およびまつり開催に伴う各種業務を効率的に管理するためのWebアプリケーションである。

初期フェーズにおいては「会員管理」を中心に据え、順次「ござ市管理」「会計管理」「通行止め管理」「衛生管理」「備品管理」「安全管理」「文書管理」「イベント管理」などの個別業務管理機能を追加拡張する。

---

## 2. 業務・機能要件

### 2.1 会員管理（フェーズ1）
実行委員会のメンバー（会員）の情報を一元管理する。

#### 2.1.1 会員属性と権限
- **複数属性の保持**: ユーザーは複数の会員属性（ロール/グループ）を重複して持つことができる。
- **会員種別の定義**:
  - **システム管理**: システム全体の管理者。仮会員の承認、ユーザーの削除、パスキーの管理、システム設定等の最上位権限を持つ。
  - **幹事**: 一般会員から選出され、幹事会への参加や意思決定権限を持つ。
  - **一般会員**: 実行委員会の基本メンバー。
  - **仮会員**: 会員登録申請直後の初期状態。システム管理者によって承認されるまで、システムの一切の機能を使用できない。
  - **その他**: 将来的に「部会担当」「事務局」などの種別が追加される可能性を考慮した設計とする。
- **権限管理**:
  - すべての機能は、システム管理者に承認されたログインユーザー（一般会員、幹事、システム管理）のみが利用可能とする。**「仮会員」はログイン可否に関わらず、一切の機能を使用できない。**
  - 会員種別および保有する属性に応じて、使用できる機能を制限する。
  - **プロフィール詳細表示**: 登録されている会員の詳細プロファイルは、「システム管理」および「幹事」グループのユーザーのみが全会員のものを閲覧でき、一般会員は自分自身の詳細情報のみが閲覧可能とするアクセス制限を敷く。

#### 2.1.2 紹介制および出自の明確化
実行委員は紹介制で加入するため、関係性を明確化する。
- メンバー登録時、**「誰の紹介か」**（紹介者である既存会員）を選択・記録する。
- 各メンバーの**「普段何をしている人か」**（職業、所属団体、得意分野など）を登録・開示できるようにし、出自や専門性を可視化する。

#### 2.1.3 除籍条件の管理
幽霊会員化を防ぐため、以下の活動実態が一切見られない会員を抽出し、除籍手続きまたは注意喚起を行うための管理フラグ・確認機能を持つ。
- LINEグループでの発言の有無
- 会議（総会・幹事会・部会）への参加状況
- まつり当日の参加状況

---

### 2.2 会議体管理およびコミュニケーション連携

#### 2.2.1 会議体の管理
システム上で以下の会議のスケジュール、出欠、および議事録を管理する。
- **幹事会**: 幹事のみが参加する会議（月1回開催）。
- **総会**: 全会員が参加可能な会議（月1回開催）。
- **部会**: 必要に応じて臨時で設置される各部門の会議。

#### 2.2.2 LINEグループとの連携・課題解決
現在LINEの専用グループで行われている運用に対し、以下のシステム的なアプローチ・支援を行う。
1. **開催案内の作成支援**:
   - 会議（幹事会、総会、部会）の開催日時やアジェンダをシステムで作成し、LINEに投稿しやすいテンプレートテキストを生成する。
2. **議事録の文章化（ホワイトボード内容の書き起こし補助）**:
   - 会議中にホワイトボードに書かれた内容（スマホ等で撮影した画像）をシステムにアップロードし、テキスト（議事録）として入力・保存・公開する仕組みを設ける。将来的にはOCRやAIによる文字起こし支援機能の追加を検討する。
3. **「この人誰？」問題の解消（LINEアカウントと実名の紐付け）**:
   - 会員プロファイル内に「LINEアカウント名（表示名）」や「LINE内でのアイコン等の特徴」を登録する項目を設ける。
   - メンバー一覧から「LINE名 ⇔ 実名」の双方向検索を可能にし、LINEグループ内での発言者が誰であるかを特定できるようにする。

### 2.3 システム共通基盤（開催年・年度管理）
本システムで扱うデータは、原則として**「開催年（年度）」に紐づく共通基盤**として設計する。これにより、今後追加されるすべての拡張機能において、年度ごとのデータが独立して管理・共有できるようにする。

#### 2.3.1 開催年（年度）の切り替えと共有
- システム全体で「現在アクティブな開催年」を保持するグローバルなコンテキストを持つ。
- 管理者はアクティブな年度を切り替えることができ、全モジュール（会員、会議、会計等）の表示・操作対象が連動して切り替わる。
- 過去の開催年を選択することで、過去ログや各種データのアーカイブ参照を可能とする。

#### 2.3.2 各機能における開催年との紐付け
- **会員管理**: 年度ごとの実行委員名簿、役職・役割分担、紹介関係。
- **会議体管理**: 年度ごとに開催される総会・幹事会・部会のスケジュールおよび議事録。
- **ござ市・会計・その他拡張機能**: 出店者情報、収支予算・実績、備品貸出履歴、イベントスケジュールなど、すべての実務データは開催年ごとに分離して蓄積する。
- **データの引き継ぎ（会員の引き継ぎ）**: 
  - 実行委員の多くは前年度から継続して引き継がれるため、新年度の立ち上げ時に前年度の会員情報（氏名、紹介者、LINE紐付け情報、所属/出自等）を一括または選択してスムーズに移行（コピー）できる機能を備える。
  - 新年度移行の際、退会者の除外や、新しい年度における役割（一般会員、幹事等）を簡単な操作で再設定・更新できるようにし、登録・移行コストを最小限に抑える設計とする。
  - 会員情報以外の基本設定（ござ市の基本設定など）についても同様に前年度からの引き継ぎを可能とする。

---

### 2.4 今後の拡張機能（フェーズ2以降）
以下の実務管理モジュールを迅速に追加可能な設計とする。これらはすべて、前述の「システム共通基盤（開催年・年度管理）」と紐づいて動作する。
- **ござ市管理**: 出店者、区画割り、出店料等の管理。
- **会計管理**: 実行委員会の予算・経費、収入・支出の管理。
- **通行止め管理**: まつり当日の道路使用許可、通行止め箇所、警備員配置等の管理。
- **衛生管理**: 出店者の検便実施状況、食品衛生管理、保健所への届出状況等の管理。
- **備品管理**: まつりで使用する各種備品（テント、机、椅子、音響機材等）の在庫、保管場所、貸出状況の管理。
- **安全管理**: 警備体制、救護所、防災計画、および緊急連絡網の管理。
- **文書管理**: 実行委員会内で作成・承認される各種申請書、届出書、マニュアル等のデジタル保管と版管理。
- **イベント管理**: まつり期間中の催し物スケジュール、出演者情報、担当スタッフの配置管理。

## 3. 技術仕様

### 3.1 開発・動作環境
- **物理サーバー（ステージング/本番）**: `192.168.11.108`（ホスト名: `reopard`）
- **ドメイン**: `www.syukuba.home`
- **開発・運用URL**: `https://www.syukuba.home`
- **データベース**: MySQL（データベース名: `syukuba_executive_comittee`）

### 3.2 アプリケーションアーキテクチャ
- **バックエンドフレームワーク**: Laravel
- **データベース**: MySQL
  - **テーブル命名規則**: データベースのすべてのテーブル名には、接頭辞として **`comittee_`** を付与する（例: `comittee_users`）。
- **フロントエンド技術**: HTML, CSS (Bootstrap), Vanilla JavaScript（プレーンなJavaScript）
  - CSSフレームワークとして **Bootstrap** を採用し、HTMLの `<link>` タグを用いて CDN から読み込む構成とする。
  - Vue.js や React などの重厚なフロントエンドフレームワークは使用しない。
  - APIの使用は最低限に抑え、**LaravelのBladeテンプレート**によるサーバーサイドレンダリングを基本とする。

### 3.3 セキュリティとアクセス制御

#### 3.3.1 ログイン認証
- **基本認証方式**: **WebAuthn (FIDO2) によるパスキー（Passkey）認証**を採用する。パスキー優先（Passkey-First）および条件付きUI（Conditional UI）設計を基本とする。
- **代替認証方式**: パスキーが利用できない環境や紛失時に備え、従来の**ID/パスワードによるログイン（およびMFA）**も併用可能とする。
- **資格情報管理**: パスキーの登録、追加、および紛失時の初期化・削除を行う「パスキー管理機能」は、**「システム管理」グループに所属するユーザーのみが操作・管理できる機能**とする。
  * 技術的制約（ユーザーのデバイスでしか秘密鍵を生成できない）を考慮し、管理者が発行した一時的な「登録セッション（ワンタイムURL）」を用いてユーザーが自身のデバイスで登録を行うフローとする。
- **詳細設計**: ログイン画面の挙動、リカバリー設計、バックエンドのセキュリティ保証レベル（LoA）等の詳細については、[login_specifications.md](file:///opt/project/syukuba-executive-committee/docs/login_specifications.md) を参照のこと。

#### 3.3.2 コンテンツセキュリティポリシー (CSP)
- **インラインコードの禁止**:
  - `Content-Security-Policy` ヘッダーに従い、HTML内に直接記述する**インラインCSS (`style="..."` や `<style>` タグ)**、および**インラインスクリプト (`onclick="..."` や `<script>` タグ内への直接記述)** の使用を全面的に禁止する。
  - スタイルおよびスクリプトはすべて外部ファイル（`.css`, `.js`）として分離し、読み込ませる。
