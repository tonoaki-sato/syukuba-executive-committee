# HTTPヘッダーインジェクション脆弱性 調査・対策報告書

## 1. 概要
本報告書は、保土ケ谷宿場まつり実行委員会 実務管理システムにおける「HTTPヘッダーインジェクション」脆弱性の有無に関する診断結果と、将来にわたるセキュリティ維持のための対策方針をまとめたものです。

---

## 2. 診断結果：脆弱性なし (安全)

コードベース全体の静的解析を行った結果、本アプリケーションには**HTTPヘッダーインジェクションにつながる脆弱性は存在しない**ことを確認しました。

### 【根拠①】PHPネイティブ `header()` 関数の非推奨・不使用
HTTPヘッダーインジェクションは、PHPネイティブの `header()` 関数に対し、改行コード（`\r` や `\n`）を含んだユーザー入力を検証せずに渡すことで発生します。
調査の結果、本システムのソースコード内では `header()` 関数は一切直接使用されていません。

### 【根拠②】Laravelフレームワーク（Symfony HttpFoundation）による自動防御
本システムでヘッダーを操作する処理（`SecurityHeaders` ミドルウェア、`DocumentController` でのPDFインライン表示レスポンスなど）は、すべてLaravel標準のResponseオブジェクト（`response()->file()` や `response()->json()` など）を介しています。
Laravelの背後で動作する Symfony の `HeaderBag` クラスには、**ヘッダー名およびヘッダー値に含まれる改行コード（`\r`, `\n`）やヌルバイト（`\0`）を自動的に検知し、除去または例外を発生させて処理を却下するフィルタリングロジックが標準で実装されています。**
このため、開発者が意識せずともフレームワークの防御壁によってヘッダーインジェクションが完全に防がれています。

### 【根拠③】セキュリティミドルウェアでの静的定義
レスポンスヘッダーを追加している唯一の共通処理である `SecurityHeaders` ミドルウェア (`app/Http/Middleware/SecurityHeaders.php`) においては、付与するヘッダー値がすべて固定文字列（静的定義）であり、ユーザーのリクエストパラメータや入力データが介入する余地はありません。

### 【根拠④】オープンリダイレクトの不在
ヘッダーインジェクションの代表的な攻撃手法として、リダイレクトヘッダー（`Location`）の制御を奪う「オープンリダイレクト」があります。
本システム内のリダイレクト処理（`AuthController`、`AdminController` など）は、すべてLaravelの名前付きルート (`redirect()->route(...)`) または `back()` による内部的な遷移先に固定されており、ユーザーが任意に指定した外部URLに動的にリダイレクトする処理は存在しません。

---

## 3. 将来的な防衛策・セキュアコーディングガイドライン

今後、機能追加（CSV/Excel出力、ファイルダウンロード、カスタムリダイレクトなど）を行う際に、ヘッダーインジェクション脆弱性を作り込まないための実装指針を定義します。

### 指針①：常にLaravelのResponse/Redirectファサードを使用する
ネイティブPHPの `header()` 関数は絶対に使用せず、Laravelのレスポンスビルダーに実装を統一してください。
* **良い例 (推奨)**:
  ```php
  return response($content)->header('Content-Type', $type);
  ```
* **悪い例 (禁止)**:
  ```php
  header("Content-Type: " . $type); // Symfonyの保護ロジックを迂回するため危険
  ```

### 指針②：動的リダイレクト時の宛先URL検証（オープンリダイレクト対策）
将来的に、パラメータで指定された遷移先にログイン後リダイレクトさせる機能を実装する場合は、リダイレクト先がドメイン内の相対パス（`/` で始まるパス）であるか、許可リストに含まれるドメインであることを必ず検証してください。
```php
$redirectUrl = $request->input('redirect_to');

// 簡易検証例: URLが '/' から始まっている（内部リンクである）ことを確認
if (str_starts_with($redirectUrl, '/')) {
    return redirect($redirectUrl);
}

return redirect()->route('dashboard'); // 不正な遷移先の場合はデフォルトにフォールバック
```

### 指針③：ファイルダウンロード時のヘッダーにユーザー入力を含める場合のサニタイズ
ユーザーがアップロードしたファイル名などをダウンロード時の `Content-Disposition` のファイル名パラメータに動的に代入する場合は、改行コードやセミコロン、ダブルクォーテーションのサニタイズ（除去またはURLエンコード）を施します。
```php
$fileName = $userUploadedFile->original_name;

// 改行コード（\r, \n）とヌルバイトを完全に除去
$safeFileName = str_replace(["\r", "\n", "\0"], '', $fileName);

return response()->download($filePath, $safeFileName);
```

### 指針④：WAF / Webサーバー（Nginx）層でのヘッダー監視
インフラ構成において、NginxやWAF（Web Application Firewall）を前段に配置し、リクエストヘッダーやリクエストパラメータに `%0d%0a` などの不審な改行エンコードが含まれるアクセスを検知・ブロックするよう設定することを推奨します。
