# 議事録登録機能の改修およびAI画像解析導入 仕様書

## 1. 背景と目的
現在、会議の議事録登録画面において、ホワイトボードの写真を複数アップロードして登録する機能が提供されています。しかし、アップロードされた写真は画像として保存されるだけであり、議事録のテキスト化は依然として手動で行われており、幹事および管理者の負担となっていました。

本改修では、以下の目的のために機能を再設計します。
1. **ホワイトボード写真の永続登録機能の廃止**: サーバーのストレージ容量の節約および管理の簡素化。
2. **AI画像解析による議事録下書き自動生成**: アップロードされたホワイトボードの写真をAI（Gemini API）で解析し、テキストとして整理した議事録下書きをフォームへ自動プリセットすることで、議事録作成作業を大幅に効率化する。
3. **一時画像の即時削除**: 解析処理に使用した画像は、処理完了後にサーバーのディスクから即座に削除し、プライバシー保護とストレージ節約を徹底する。

---

## 2. 変更内容一覧

| 機能エリア | 変更前仕様 | 変更後仕様 |
| :--- | :--- | :--- |
| **データベース** | `comittee_meetings` の `whiteboard_images` カラムに画像パスの配列をJSON保存 | カラムをマイグレーションにて削除し、既存のアップロード画像もサーバー上からすべて削除する |
| **会議詳細画面** | 登録済みのホワイトボード写真一覧を表示する | ホワイトボード写真の表示エリアを削除する |
| **議事録編集画面** | 複数のホワイトボード写真をアップロード・保存するフォームを表示 | ホワイトボード写真の追加フォームおよび登録済み表示を削除。<br>代わりに、**「AIで議事録下書きを生成」**エリアを追加。 |
| **AI解析機能** | なし | 画像をアップロードして非同期でAI解析を実行し、テキストエリアに結果をプリセットする。解析後の画像は即時削除する。 |

---

## 3. 詳細設計

### 3.1 データベース設計
既存の `comittee_meetings` テーブルにある `whiteboard_images` カラム（JSON型、nullable）は、**マイグレーションを新設してテーブルから完全に削除します**。

#### ① カラム削除マイグレーションの作成
新規マイグレーションを作成し、`comittee_meetings` テーブルから `whiteboard_images` カラムをドロップします。
```php
Schema::table('comittee_meetings', function (Blueprint $table) {
    $table->dropColumn('whiteboard_images');
});
```

#### ② 過去データの削除（物理ファイルのクリーンアップ）
過去にアップロードされ、`storage/app/public/whiteboards/` ディレクトリに保存されている既存のホワイトボード画像ファイルについても、サーバー上から完全に削除します。
マイグレーションの `up` メソッド内で、ディレクトリ内のファイルをすべて削除するロジック（`File::deleteDirectory(storage_path('app/public/whiteboards'))` など）を実行し、ディスク空き容量を確保します。

### 3.2 画面設計 (UI/UX)

#### ① 会議詳細画面 (`resources/views/meetings/show.blade.php`)
- 104〜125行目の「📷 ホワイトボードの写真（画像アーカイブ）」のHTMLセクションを**完全に削除**します。

#### ② 議事録編集画面 (`resources/views/meetings/minutes.blade.php`)
- 現行の「📷 ホワイトボード写真 of 追加」フォーム（38〜43行目）および「現在登録済みの画像」表示セクション（46〜59行目）を**削除**します。
- 議事録本文のテキストエリア（`#minutes`）の直上に、新しく**「📷 ホワイトボード画像から議事録下書きを生成 (AI)」**エリアを追加します。
  - **構成要素**:
    - ファイル選択インプット: `id="whiteboard_image_ai"`, `accept="image/*"`（単一ファイルのみ）
    - 実行ボタン: `id="btn-analyze-whiteboard"`, 表示テキスト: `AI解析を実行して下書きを作成`（初期状態は無効化、ファイル選択時に有効化）
    - ローディング表示: 解析中に「画像を解析中...（最大30秒程度かかる場合があります）」といったスピナーとメッセージを表示するエリア
  - **セキュリティ・アセット構成（SCP/CSP対応）**:
    - **インライン実装の禁止**: セキュリティ方針（SCP）に基づき、Bladeテンプレート内への `<script>` および `<style>` タグのインライン記述、タグの `style` 属性、`onclick` や `onchange` 等のインラインイベントハンドラ属性の使用を**全面的に禁止**します。
    - **JavaScriptの外部化**: 解析処理のJavaScriptは、新規ファイル `resources/js/meetings/minutes-ai.js` に記述し、`vite.config.js` に登録した上で、Bladeテンプレート側からは `@vite('resources/js/meetings/minutes-ai.js')` を用いて読み込みます。
    - **CSSのクラス化**: UI上のスタイル調整（非表示クラスや装飾）は、Bootstrap 5のユーティリティクラス（`d-none`, `text-muted` 等）を優先的に使用し、必要に応じて `resources/css/app.css` にスタイルを定義してクラス指定で適用します。

  - **フロントエンド制御 (JavaScript - `minutes-ai.js`)**:
    - DOM要素のイベントリスナー（`addEventListener`）を介してファイルの選択状態やボタンのクリックイベントを監視します。
    - ユーザーが画像を選択し、「AI解析を実行して下書きを作成」をクリックした際、`fetch` API等を用いて非同期でサーバーへ送信します。
    - 送信中（解析中）は、二重送信を防ぐため「実行ボタン」および「議事録保存ボタン」を `disabled` にします。
    - **プリセット処理**:
      - 解析に成功した場合、取得した議事録下書きテキストを `textarea#minutes` に挿入します。
      - もし `textarea#minutes` にすでに文字が入力されている場合は、確認ダイアログ（例: `「すでにテキストが入力されています。AIが生成した下書きで上書きしてもよろしいですか？」`）を表示し、承認された場合のみ上書きします。
      - 解析失敗時は、レスポンスから取得したエラーメッセージをアラートで表示します。

### 3.3 バックエンド設計

#### ① ルーティングの追加 (`routes/web.php`)
幹事および管理者のみがアクセスできるミドルウェアグループ内に、AI解析用のエンドポイントを追加します。
```php
Route::post('/meetings/{meeting}/minutes/analyze', [MeetingController::class, 'analyzeWhiteboard'])->name('meetings.minutes.analyze');
```

#### ② モデルの修正 (`App\Models\Meeting.php`)
- `$fillable` 配列から `'whiteboard_images'` を削除します。
- `$casts` 配列から `'whiteboard_images' => 'array'` を削除します。

#### ③ コントローラー処理 (`App\Http\Controllers\MeetingController.php`)
- `updateMinutes` メソッドから、古い `whiteboard_images` のバリデーション（`whiteboard_images`, `whiteboard_images.*`）および画像のアップロード・保存処理を完全に削除します。
- 新規アクション `analyzeWhiteboard` を実装します。
  - **バリデーション**:
    - `image`: 必須, 画像形式（`jpeg,png,jpg,webp`）, 最大10MB (`max:10240`)
  - **処理フロー**:
    1. 送信された画像ファイルを、サーバー内の一時ディレクトリ（例: `storage/app/tmp`）に一時的に保存します。
    2. 一時保存した画像ファイルからバイナリデータを取得し、Base64エンコードします。
    3. `Http` クライアントを使用して Gemini API にリクエストを送信します（詳細は 3.4 節）。
    4. APIからのレスポンスを解析し、下書きテキストを抽出します。
    5. **【重要・必須】APIの呼び出しが成功したか失敗したかにかかわらず、`finally` ブロック等で一時保存した画像ファイルを必ずディスクから削除します。**
    6. 成功時は解析結果テキストを、失敗時はエラーメッセージを含めた JSON レスポンスを返却します。

### 3.4 AI解析（Gemini API）連携設計

#### ① 接続仕様
- **APIエンドポイント**: 
  `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent?key={API_KEY}`
- **使用モデル**: `gemini-2.5-flash`
  - 理由: 高速であり、手書きの画像解析（マルチモーダル処理）に優れているため。
- **タイムアウト設定**: 画像解析には時間がかかる場合があるため、HTTPクライアントの接続タイムアウトは `30秒` 以上に設定します。

#### ② 設定値の管理
- `.env` に API キーを設定します。
  ```env
  GEMINI_API_KEY=YOUR_GEMINI_API_KEY_HERE
  ```
- `config/services.php` に定義を追加し、コード内からは config 経由で呼び出します。
  ```php
  'gemini' => [
      'key' => env('GEMINI_API_KEY'),
  ],
  ```

#### ③ 送信プロンプト設計
画像とともに送信するテキストプロンプトは以下のように設計します。
> 添付されたホワイトボードの写真を詳細に解析し、会議の議事録の下書きとして整理してください。
> 手書き文字をテキスト化し、議論の流れや決定事項、宿題事項（タスク）が分かるようにまとめてください。
> 出力は分かりやすい日本語の文章または箇条書きとし、HTMLタグを含めず、Markdown形式で整理して出力してください。

#### ④ レスポンスのパース処理
Gemini API の返却する JSON 構造から、以下のパスでテキストを取得します。
- `candidates[0].content.parts[0].text`
テキストが空である場合や構造が想定と異なる場合は、エラーハンドリングを行います。

### 3.5 エラーハンドリング
AI解析機能において、以下のエラーケースを想定し、それぞれ適切なハンドリングを行います。

1. **APIキー未設定エラー**:
   - `GEMINI_API_KEY` が設定されていない場合、サーバー側で検知して `500` エラー（メッセージ: 「AI解析機能が設定されていません。管理者にお問い合わせください。」）を返します。
2. **タイムアウト・通信エラー**:
   - APIとの通信中にタイムアウトまたは接続エラーが発生した場合、例外をキャッチし `500` エラー（メッセージ: 「AIサーバーとの通信中にタイムアウトが発生しました。時間をおいて再度お試しください。」）を返します。
3. **画像解析エラー**:
   - 画像が不鮮明で解析できない、またはサポート外の画像である場合、Geminiからのエラーレスポンスを検知し `500` エラー（メッセージ: 「画像の解析に失敗しました。文字が鮮明に写っているか確認してください。」）を返します。
4. **一時ファイル削除の保証**:
   - `try-catch-finally` 構文を用い、如何なる例外が発生した場合でも、サーバーにアップロードされた画像の一時ファイルが確実に削除されるようにします。
