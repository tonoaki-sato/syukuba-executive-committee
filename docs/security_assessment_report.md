# セキュリティ評価・脆弱性診断報告書

## 1. エグゼクティブサマリー
保土ケ谷宿場まつり実行委員会 実務管理システムにおけるセキュリティ設計および実装の評価を行いました。

本システムは、パスワード漏洩リスクを根本から排除する**完全パスワードレス認証（WebAuthn / パスキー）**の導入、グローバルなCSRF保護、および `SecurityHeaders` ミドルウェアによる主要なセキュリティヘッダー（HSTS, CSP, X-Frame-Options 等）の適用がなされており、全体として**非常に高いセキュリティ基準**で構築されています。

一方で、現状のセキュリティ設定とアセット実装の不整合による不具合の懸念や、多層防御（Defense in Depth）の観点から改善が推奨される点がいくつか見つかりました。以下に診断詳細と具体的な改善策を報告します。

---

## 2. 診断と評価結果

| 評価項目 | 診断結果 | 評価と根拠 |
| :--- | :--- | :--- |
| **SQLインジェクション** | **安全 (脆弱性なし)** | クエリ構築にはEloquent ORMおよびクエリビルダが徹底されており、安全にパラメータバインドされています。唯一生のSQLを使用している `EquipmentMatrixController` の `orderByRaw` も、静的な文字列のみで構成されており安全です。 |
| **CSRF (クロスサイト要求偽造)** | **安全 (脆弱性なし)** | `bootstrap/app.php` にてグローバルにCSRF検証ミドルウェアが適用されています。フォームやFetch通信時のトークン（`X-CSRF-TOKEN`）付与も徹底されています。 |
| **XSS (クロスサイトスクリプティング)** | **安全 (脆弱性なし)** | Bladeテンプレートの基本エスケープ（`{{ }}`）が使われており、エスケープなし出力（`{!! !!}`）の利用は凡例の絵文字出力などの静的データのみに限定されています。 |
| **ヘッダーインジェクション** | **安全 (脆弱性なし)** | PHPネイティブの `header()` 関数は一切使用されておらず、レスポンスの生成はすべてSymfonyの自動改行コード除去フィルターが機能するLaravel Responseファサードに統一されています。 |
| **アクセス制御 (認可制御)** | **安全 (脆弱性なし)** | 一般会員と幹事・管理者の権限分離は、各コントローラーの処理冒頭で `isSystemAdmin()` や `isKanji()` 判定を用いて厳格にガードされています。 |

## 3. 指摘事項と具体的な改善策

サイト全体の解析により判明した、セキュリティ設定上の不整合および改善推奨ポイントです。

### 指摘①：CSP (Content Security Policy) とインラインスクリプトの不整合（重要・不具合の懸念）
* **現状の設計**:
  `SecurityHeaders` ミドルウェアにおいて、CSPの `script-src` に `'self' https://cdn.jsdelivr.net` が指定されていますが、**`'unsafe-inline'` が許可されていません。**
* **問題点**:
  以下の複数のBladeビュー内に、インライン形式（`<script>` タグ直接記述）のJavaScriptが存在します。
  - `goza/applications/create.blade.php` (動的フォーム表示制御)
  - `goza/applications/edit.blade.php`
  - `goza/map/index.blade.php` (地図ピン操作関連)
  これにより、ブラウザのセキュリティ機能（CSP）によって**これらのインラインスクリプトの実行が強制ブロックされ、画面の動的機能（条件分岐フォームの表示等）が動作しない不具合が発生しています。**
* **改善策（以下のいずれか）**:
  1. **（推奨）アセットの外部化**: インラインスクリプトを外部JSアセットファイル（例: `public/js/goza/applications.js` 等）に分離し、インラインコードを完全に排除する（本システムのセキュアコーディング規約にも合致）。
  2. **CSP Nonceの導入**: リクエストごとにランダムな一時トークン（Nonce）を生成してミドルウェアでCSPヘッダーに含め、Blade側の `<script nonce="{{ csp_nonce() }}">` に付与して動的に実行を許可する。

---

### 指摘②：幹事・管理者向けルーティングの二重ガード不足（多層防御の改善点）
* **現状の設計**:
  `routes/web.php` における「幹事 ＆ 管理者ルート（会議作成・議事録等）」のミドルウェアグループには、`['auth', 'approved']` のみが指定されています。
* **問題点**:
  コントローラー側（`MeetingController`）の各メソッドの冒頭で `Auth::user()->isKanji()` などの権限チェックが入っているため、一般ユーザーが実行・閲覧することはできません。しかし、ルーティングのレベルでは保護されていないため、一般会員でもURLの叩き込み（例: `/meetings/create`）によりアクションまで到達してしまいます。
  コントローラー側のチェック漏れや将来的なバグ（チェックの消し忘れ）により権限昇格が発生するリスクがあります。
* **改善策**:
  `routes/web.php` でこれらのルートグループに対し、幹事・管理者限定のミドルウェアエイリアス（現在 `EnsureUserIsKanjiOrAdmin` クラスに割り当てられている `gozaichi` ミドルウェア）を適用し、URLの到達前段階でブロックする多層防御（二重ガード）を確立してください。
  ```php
  // web.php の該当箇所を gozaichi (または新設の kanji-admin) ミドルウェアで保護
  Route::middleware(['auth', 'approved', 'gozaichi'])->group(function () {
      Route::get('/meetings/create', [MeetingController::class, 'create'])->name('meetings.create');
      ...
  });
  ```

### 指摘③：パスキー再登録セッション発行時の「既存キー即時削除」の仕様（可用性のリスク）
* **現状の設計**:
  `AdminController@createPasskeySession` にて、管理者が既存アクティブ会員に対してパスキーの再発行セッション（ワンタイムトークン）を発行した際、**その時点で対象ユーザーの登録済みパスキーを即座にすべて削除しています**。
  `WebAuthnKey::where('user_id', $user->id)->delete();`
* **問題点**:
  トークンを発行した瞬間から、ユーザーが実際に新しいパスキーの登録設定を完了するまでの間、該当ユーザーはシステムに一切ログインできなくなります。もし管理者が誤ってボタンを押した場合や、ユーザーが24時間以内に設定を完了しなかった場合、即座に締め出されてしまいます（可用性の欠如）。
* **改善策**:
  再発行セッションを発行した段階では既存のパスキーは削除せず、ユーザーがワンタイムURLにアクセスして新しいデバイスのパスキー登録（認証）に**成功したタイミングで、古いパスキーを削除（または上書き）する**仕様に変更することを推奨します。これにより、移行期間中のログイン不能状態を防ぐことができます。
