# 保土ケ谷宿場まつり実行委員会 実務管理総合システム 会員プロフィール編集機能 仕様書

本書は、会員（実行委員）のプロフィール情報を変更するための「プロフィール編集機能」の要件および詳細仕様を定義する。

---

## 1. 機能概要

会員情報の実態（所属、得意分野、LINEアカウント名等）の変更に柔軟に対応し、名簿の最新状態を維持するため、以下の2つの編集フローを提供する。

1. **一般ユーザーによる自己編集フロー**:
   * ログイン中の会員が、自身のマイページから連絡先や職業、スキルなどのプロフィール情報を編集する。
   * 管理者権限が必要な項目（ロールやステータス）は変更できない。
2. **システム管理者による他会員の代理編集フロー**:
   * システム管理グループ（管理者）が、全会員の詳細プロフィール画面から、対象会員のすべての情報（ロールやステータスを含む）を編集する。

---

## 2. 権限・アクセス制御（編集可能項目の区分）

役割に応じて、編集・更新できる項目を制限する。

| 項目名 | カラム名 | 一般会員による自己編集 | 管理者による編集 | 備考 |
| :--- | :--- | :---: | :---: | :--- |
| **氏名（漢字）** | `name` | ○ | ○ | 最大50文字 |
| **氏名（かな）** | `name_kana` | ○ | ○ | ひらがなのみ |
| **メールアドレス** | `email` | ○ | ○ | ログインIDを兼ねる。重複不可 |
| **本業・職業** | `profession` | ○ | ○ | 出自の明確化のため必須 |
| **所属団体・会社名** | `affiliation` | ○ | ○ | 任意入力 |
| **得意分野（スキル）** | `skills` | ○ | ○ | チェックボックス（複数選択） |
| **LINEアカウント名** | `line_display_name` | ○ | ○ | LINE内表示名。必須 |
| **紹介者（既存会員）** | `referrer_id` | × （表示のみ） | ○ | 管理者のみ紐付け変更可能 |
| **紹介者（テキスト）** | `referrer_text` | × （表示のみ） | ○ | 管理者のみ紐付け変更可能 |
| **会員ロール（役割）** | `roles` | × （非表示） | ○ | 一般 / 幹事 / システム管理の割り当て |
| **在籍ステータス** | `status` | × （非表示） | ○ | active / suspended / expelled / temporary 等 |
| **承認者 / 承認日時** | `approved_by` / `approved_at` | × （表示のみ） | × （表示のみ） | システムによる記録値のため編集不可 |

## 3. 画面・遷移設計

### 3.1 一般ユーザー自己編集画面
- **編集の開始**: マイページ画面（`/mypage`）に「プロフィールを編集する」ボタンを追加。
- **編集画面ルート**: `mypage.edit` (`GET /mypage/edit`)
- **更新処理ルート**: `mypage.update` (`POST /mypage/update` または `PUT /mypage`)
- **アクセス制御**: ログイン中かつ承認済みのユーザーのみ。

### 3.2 管理者用代理編集画面
- **編集の開始**: 全会員の詳細プロフィール画面（`/users/{user}`）の管理者アクションエリアに「会員情報を編集する」ボタンを追加。
- **編集画面ルート**: `admin.users.edit` (`GET /admin/users/{user}/edit`)
- **更新処理ルート**: `admin.users.update` (`POST /admin/users/{user}/update` または `PUT /admin/users/{user}`)
- **アクセス制御**: システム管理グループ（`admin` ロール）のみ。

## 4. バックエンド処理・データ保存仕様

プロフィール編集情報が送信された際、以下の処理を行う。

1. **バリデーションチェック**:
   - 送信された情報が型、必須条件、および文字数制限を満たしていることを検証する。
   - `email` については、**自身の現在のメールアドレスを除外した**ユニーク制約チェック（`unique:comittee_users,email,{$id}`）を適用し、他ユーザーとの重複のみをエラーとする。
   - `name_kana` はひらがなのみ（`regex:/^[ぁ-んー ]+$/u`）であることを検証する。
2. **データベースの更新**:
   - 対象の `comittee_users` レコードの各カラムを更新する。
   - 管理者による編集で `roles` が変更された場合、**当年度の在籍レコード (`comittee_user_years`）の `roles` カラムも連動して自動更新**する。
3. **結果の返却**:
   - 自己編集完了後は、マイページ（`/mypage`）へリダイレクトし、フラッシュメッセージ「プロフィールを更新しました。」を表示する。
   - 管理者による編集完了後は、会員詳細画面（`/users/{user}`）へリダイレクトし、フラッシュメッセージ「会員情報を更新しました。」を表示する。

---

## 5. セキュリティ・アクセス制御

- **不正アクセス防止**:
  - 自己編集ルートにおいて、他人のユーザーIDをパラメータとして指定して更新することはできない設計とする（認証ユーザー `Auth::user()` のオブジェクトに対して直接更新処理を行う）。
  - 管理者用編集ルートは `EnsureUserIsSystemAdmin` ミドルウェアによって保護し、管理者以外のアクセスには `403 Forbidden` を返却する。
- **CSP（コンテンツセキュリティポリシー）準拠**:
  - 編集画面のHTML/Bladeテンプレート内にはインラインスクリプトおよびインラインCSS（`style` や `<script>` タグ）を一切含めず、動的制御が必要な場合は外部JSファイルを介して行う。
